5 Whys Analysis

System analyzed: Same as Task 3 (UALR registration).
Problem statement: During registration peaks, students experience severe slowness and timeout errors that prevent successful enrollment.

5 Whys (Bandwidth Focus)

  1. Why can students not complete registration during peak times? Because the registration pages and submit calls time out under heavy traffic.
  2. Why do the pages and submit calls time out? Because available network/server capacity is saturated when many users submit actions simultaneously.
  3. Why is capacity saturated during those windows? Because the system is sized for average daily traffic, not short enrollment spikes with high concurrency.
  4. Why is the system sized for average traffic instead of peak enrollment traffic? Because peak-event forecasting, load-testing cadence, and burst-capacity planning are not integrated into term readiness procedures.
  5. Why are forecasting and burst-capacity planning not integrated into readiness procedures? Because ownership is split across departments (academic scheduling, IT operations, and vendor/platform support) without a single performance governance process and clear service thresholds.
Root Cause Summary: The core issue is not only raw bandwidth; it is the absence of a cross-functional peak-readiness process that translates expected enrollment demand into tested and provisioned capacity.

Targeted Corrective Actions

  1. Create a pre-registration readiness checklist with explicit capacity thresholds and sign-off owners.
  2. Run recurring load tests before each registration cycle using realistic student behavior patterns.
  3. Define automatic burst-scaling or queue rules for the registration window.
  4. Set monitoring alerts for response time, error rate, and queue depth with clear escalation paths.
  5. Perform post-registration retrospectives and feed lessons into the next cycle.
