1. Summary

Daniel introduced process modeling as a distinct systems analysis technique — separate from root cause analysis — for visualizing and understanding information and business processes. He surveyed several modeling methods (flowcharts, swim lane diagrams, Gantt charts, BPMN) before deep-diving into Data Flow Diagrams (DFDs). DFDs use just four symbol types to model data movement inside IT systems. He walked through a student registration example, demonstrated leveling (decomposition), and live-edited Wikipedia's DFD article during class.

2. Process Modeling in Systems Analysis

Process modeling fits within a broader toolkit of systems analysis techniques:

TechniquePurpose
Root Cause AnalysisIdentify why problems occur
Process ModelingVisualize & formalize information/business processes
BenchmarkingMeasure how well organizational processes work
Test Solutions / PrototypingBuild throwaway solutions to learn, then build the real system
Cost-Benefit AnalysisEnsure benefits merit the costs
AutomationDecide what to automate (emerging: AI vs. human decision)
Lean Six SigmaToolbox for various systems analysis activities
User InvolvementGet end users involved in design (contrast: Blackboard)
"Systems analysis is about specifying and designing improved IT activities in an organization." Process modeling helps you visualize those processes so you can communicate about them, identify improvements, and understand them better.

3. Flowcharts

The earliest formalized process modeling method.

History

  • First standardization: "Operation and Flow Process Charts" (1947)
  • Published by the American Society of Mechanical Engineers — not computer engineers
  • Flowcharts were invented in a world not even thinking about computers

Standard Symbols

ShapeMeaning
OvalStart / End
RectangleProcessing
ParallelogramInput / Output
DiamondDecision (if/else branching)

Example: Robotic Vacuum Cleaner

  1. Check: Has the vacuum finished its route? (diamond)
  2. If yes → turn off
  3. If no → check: Is the battery low? (diamond)
  4. If battery low → turn off (even though route is unfinished)
  5. If battery OK → continue vacuuming

4. Swim Lane Diagrams

Named because they look like a swimming pool with lanes. Each lane (column or row) represents a different role in the process.

Medical Setting Example

LaneActivities
PatientArrives at hospital
NursePrepare patient → Instruct patient → Discharge patient
DoctorTreats the patient

Arrows show flow of control between lanes.

Best practice: Limit swim lane diagrams to 4–6 lanes. More than that becomes too complex to understand. Two classification approaches: by domain (who does what) and by task (what gets done).

Fun moment: Daniel displayed a swim lane diagram written in Telugu script and had an AI translate it.

5. Gantt Charts

Horizontal bar charts showing timelines of different project sub-parts.

  • Show when tasks start and end
  • Reveal dependencies — e.g., Task B can't begin until Task A completes
  • Some tasks may overlap slightly
  • Different colors can indicate categories
  • Most students had seen Gantt charts before, even if not by name

Will be covered in more depth later in the course.

6. BPMN (Business Process Model and Notation)

A standardized set of graphical symbols for modeling business processes, with free and paid drag-and-drop tools available.

Daniel's anecdote: He and a PhD student submitted a paper with a custom block model. A reviewer said: "Block models are great, but you should use a standard method like BPMN." Since Daniel teaches BPMN in this class, he went and converted it — proving the value of using standard languages that communicate well with others rather than inventing your own notation.

Fun moment: Daniel showed a BPMN description in Arabic, and a student who could partially read it helped translate the technical terms.

7. Data Flow Diagrams (DFDs) — Main Topic

A DFD is a visual, graphical model of how data flows, is transformed, and is stored within an IT system (or even objects in a factory).

The Four Symbol Types

SymbolShapePurposeExamples
External Entity Sharp-cornered rectangle Inputs & outputs — not part of the system Customers, students, faculty, suppliers, outside IT systems (e.g., banking system at a retail store)
Process Rounded-corner rectangle (or circle) Transforms / processes data — "where the action happens" Enroll student, build schedule, produce class list
Data Store Parallel lines (open or closed on one end) Any data repository Database, file, data warehouse
Data Flow Labeled arrow Shows movement of data between components "Enrollment request," "class list," "schedule"
Key limitation: DFDs cannot express decisions (no diamond/if-else symbol). If decision-making is central to what you're modeling, use flowcharts instead. DFDs constrain how you think about the system — they focus purely on data movement and transformation.

"You can do an awful lot with four symbols, believe it or not."

8. Student Registration DFD Example

Daniel walked through a complete data flow diagram for registering a student in a course.

External Entities (rectangles)

  • Academic Department (input) — provides offered courses
  • Faculty Member (output) — receives class list
  • Student (input & output) — sends enrollment requests, receives schedule

Data Stores

  • Offered courses (for the semester)
  • Course enrollment records
  • Student database

Processes (numbered)

#ProcessWhat It Does
1Build Course ScheduleTakes offered courses → determines classroom, times, etc.
2Enroll StudentsTakes enrollment requests, course info, student data → produces schedules and enrollment records. Most complex — prime candidate for leveling.
3Produce Class ListCombines data from all 3 stores → sends class list to faculty

9. Leveling (Decomposition)

Leveling means taking a single process from a DFD and creating a new, more detailed DFD showing its internal sub-processes.

The Hierarchy

LevelNameDescription
Level 0Context DiagramOne single process representing the entire system, with all external entities and data stores connected to it
Level 1Main DFDThe context process is broken into numbered sub-processes (1, 2, 3…)
Level 2Detailed DFDA single process (e.g., Process 2) is decomposed into 2.1, 2.2, 2.3…
Level 3+Further detailProcess 2.3 can be decomposed into 2.3.1, 2.3.2, 2.3.3…
"You can take a data flow diagram and treat it like a node in a tree, and then below that, you'll have detailed diagrams for each of the processes."

Context Diagram for Student Registration

At the top level, a single process called "Register Students" would receive input from the Academic Department and produce outputs to Students and Faculty — all the internal processes (scheduling, enrollment, class lists) are bundled inside.

Homework connection: The homework asks you to (1) create a DFD, then (2) pick one process and level it by creating a second, more detailed DFD for that process's internals.

11. Class Q&A Review

Daniel quizzed the class on DFD components at the end:

  • Arrows — show steps in the flow of data
  • Data flows to: processes, data stores, or external entities
  • Student mentioned "decisions" → Daniel clarified that DFDs do not have a decision symbol; that's a flowchart concept

12. Key Takeaways

  1. Process modeling formalizes system understanding through visual diagrams
  2. Multiple methods exist (flowcharts, swim lanes, Gantt, BPMN, DFDs) — each with trade-offs
  3. DFDs are deliberately simple (4 symbols) but powerful for modeling data movement
  4. Leveling lets you zoom into any process for more detail, creating a hierarchy
  5. The context diagram is the highest-level view (one process, all I/O)
  6. DFDs cannot model decisions — use flowcharts for that
  7. Always use standard notations rather than inventing your own
