2. Core PHP Concept: Server vs Browser Execution

00:00-16:00 (main recording)

The class centered on one key concept: PHP runs on the server, while JavaScript runs in the browser.

LanguageWhere It RunsWhat Browser Sees
PHPServer-side (before response is sent)Only generated HTML output
JavaScriptClient-side (inside browser)Script code plus runtime behavior
Daniel emphasized the "three-view model": rendered page, browser view-source, and true server file source. Understanding differences across these views explains most PHP confusion.

3. PHP Output and File Extension Behavior

16:00-33:00

Echo as Output

  • echo is PHP's output command.
  • PHP can output plain text or HTML markup such as italics tags.
  • No echo means no visible output, even if computation occurs.

Why .php Matters

  • A file ending in .php signals server execution of PHP code blocks.
  • Renaming the same file to .html stops PHP execution.
  • When PHP is not executed, browsers may treat PHP syntax as unknown tag-like text and render confusing output.
If you want PHP behavior in your project pages, the file suffix must be .php. Otherwise, the server ships raw content without running PHP logic.

4. Can JavaScript Add PHP?

33:00-42:00

Daniel addressed a common trap: JavaScript can insert text that looks like PHP, but this does not execute as PHP.

  • JavaScript runs after page delivery in the browser.
  • PHP execution already happened earlier on the server.
  • So inserted PHP-like text is just text/markup from the browser's perspective.

5. Syntax Notes: Comments in PHP vs JavaScript

56:00-end
  • // works for single-line comments in both PHP and JavaScript.
  • /* ... */ works for multi-line comments in both.
  • # is valid as a PHP comment style but not standard JavaScript comment syntax.

The class used short exercises to reinforce comment syntax and how comments help isolate code while debugging.
