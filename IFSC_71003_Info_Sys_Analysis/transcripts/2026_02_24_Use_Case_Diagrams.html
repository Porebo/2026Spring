Lecture Summary: Use Case Diagrams

Module 3 | Feb 24, 2026 | 2:54 PM – 3:47 PM

1. Narrative of Activities

Professor Daniel Berleant introduced use case diagrams and their role in software engineering. The lecture built on prior discussions of process modeling (data flow diagrams, BPMN) and shifted focus to how systems are described from the user's perspective. The class worked through examples — elevators, text editors, and restaurants — to develop intuition about use case granularity. The session concluded with AI-generated true/false and multiple-choice questions that the class answered together.

2. What Is a Use Case?

  • A use case is a description of how a user interacts with a system — an example of how a system is used.
  • It is not inherently a diagram. A use case is a concept that can be expressed as a diagram or as written text.
  • A use case diagram is the visual representation using UML-style notation (stick figures for actors, ovals for use cases, a rectangle for the system boundary).

Systems That Use Cases Can Describe

  • Software applications, websites, business processes
  • Physical systems like cars, elevators
  • Entire businesses (e.g., an online store's checkout flow)
  • Originally a software engineering technique (translated from "usage case")

3. Sea Level vs. Fish Level (Cockburn's Framework)

Alistair Cockburn is widely regarded as the top authority on use cases.

LevelNameMeaningFocus
HighSea levelHolistic, goal-orientedWhat the user wants to accomplish
LowFish levelGranular, step-by-stepIndividual interactions with the system
  • Sea level = the system boundary (the box); views the entire interaction as one purpose
  • Fish level = the individual ovals inside the box; each step is its own use case
  • Granularity is a design decision — neither level is inherently correct

When to Use Each

  • Sea level → Communicating with users/stakeholders about what they want
  • Fish level → Designing the internal structure of a system

4. The Elevator Example

The class examined a use case diagram for an elevator with two ovals: "Press Outside Button" and "Press Floor Button."

Key question: Is this one use case or two?

  • Two use cases (fish level): Each button press is a separate interaction with the system
  • One use case (sea level): The user's goal is simply to get from one floor to another; pressing buttons are steps within that single goal

Both interpretations are valid — it depends on the chosen level of granularity.

5. The Restaurant Example

A more complex use case diagram with multiple actors (waiter, client, cashier, chef) and ~9 use cases (order food, serve food, cook food, eat food, order wine, serve wine, drink wine, pay for food, pay for wine).

  • Even at sea level, this system still has multiple use cases
  • Example groupings: wine-related actions together, food-related actions together, payment actions together

6. Use Case Diagrams Do NOT Show Sequence

  • The restaurant diagram doesn't indicate that food must be ordered → cooked → served → eaten → paid for
  • Top-to-bottom reading is not reliable for determining order
  • If you need sequence, use a written use case (textual description listing steps in order)
  • A use case diagram is higher level and more abstract — it intentionally omits sequence information

8. In-Class Practice: True/False Questions

  • "A use case describes how a user interacts with a system to achieve a goal."
    → Debatable. True for sea-level use cases, but fish-level ones may not be goal-oriented. Not strictly true in all cases.
  • "A use case diagram shows the exact sequence of steps performed."
    → False. Diagrams show actors, goals, and relationships — not step-by-step order.
  • "Sea-level use cases are more granular and detailed than fish-level use cases."
    → False. Fish level is more granular. Sea level is goal-oriented.
  • "A written use case typically includes a sequence of actions."
    → True. Writing them down allows you to capture sequence.
  • "Choosing the granularity of a use case is a design decision."
    → True. It's a modeling choice made by the designer.

9. Key Authorities

  • Alistair Cockburn — Top authority on use cases; sea level vs. fish level framework
  • Stephen Schach — Software engineering textbook author; advocates for single (goal-oriented) use cases

10. Key Takeaways

  • A use case is a concept, not just a diagram
  • Granularity (sea vs. fish) is a design choice with no single correct answer
  • Use case diagrams are great for brainstorming and high-level understanding
  • For sequence and detail, use written use cases
  • Real systems require thinking about missing elements (alternative paths, other actors, edge cases)
