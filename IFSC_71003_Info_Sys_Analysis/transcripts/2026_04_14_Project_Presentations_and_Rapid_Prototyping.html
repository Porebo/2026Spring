2. Final Project Presentation Logistics

Main administrative focus
  • The last two class meetings would be used for project presentations.
  • Each student would get roughly five to six minutes to walk through the project website.
  • Daniel planned to load the website on his own screen while the student narrated what to click and what to notice.
  • Students could talk about implementation, source code, design decisions, and future expansion ideas.
The presentation model was intentionally lightweight: no slide deck required, just a clear walkthrough of the working site.

3. Submission and Attendance Expectations

Presentation details
  • The project needed a prominent link from the student's personal site.
  • Students were asked to email Daniel the exact name they wanted displayed on the project-link page.
  • Group members would still present individually, though Daniel would try to schedule them back-to-back.
  • U-section students could submit a recorded walkthrough instead of attending live, while live-section students could present either in person or online.

4. Backup and Hosting Risk Reminder

Practical warning
Daniel strongly advised students to keep frequent backups of their project files. Free hosting can suspend sites unexpectedly, especially during periods of heavy editing or shared team activity.

This was positioned as a real end-of-semester risk management habit, not just a generic caution.

5. HW12 PHP Deliverables

Assignment walkthrough
  • Add three new PHP commands to the demo page.
  • Quote the PHP code on the demo page.
  • Link to that demo page from the homework answer page.
  • List the three commands that were added.
  • Add a web page hit counter, which Thursday's class would explain in detail.

6. Rapid Prototyping Framing

Brief concept intro

Daniel described a rapid prototype as a system front end that looks like the finished product without yet containing the full technical back end.

  • It can imitate the final user experience.
  • It usually omits databases and more advanced algorithms.
  • Its value is in look-and-feel, concept communication, and early feedback.
Students were asked to connect this idea directly to their own course project and submit a screenshot-based written response rather than build a separate heavy artifact.

7. Overall Class Message

  1. Project readiness now mattered more than perfecting isolated mini-exercises.
  2. Students needed working links, visible progress, and presentation readiness.
  3. Rapid prototyping was introduced mainly as a lens for thinking about their existing project work.
