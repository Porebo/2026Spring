3. HTML Form Lifecycle

Form ? PHP Round Trip

Using the W3Schools sandbox, Daniel dissected:

  • <form action="/html/action_page.php"> — the action attribute sends data to a PHP program on the server.
  • Query-string output: submitting firstname=John, lastname=Doe produces ?firstname=John&lastname=Doe.
  • Manually visiting https://www.w3schools.com/action_page.php shows the PHP response; appending ?foo=bar echoes arbitrary parameters.
  • Security constraints block submissions from outside W3Schools (e.g., InfinityFree), illustrating cross-origin protections.
<form action="/html/action_page.php"> <label for="fname">First name:</label> <input type="text" id="fname" name="firstname" value="John"> <br><br> <label for="lname">Last name:</label> <input type="text" id="lname" name="lastname" value="Doe"> <br><br> <input type="submit" value="Submit"> </form>

4. Labels, Inputs & Accessibility

  • label elements use for to bind to inputs with matching id; clicking the label focuses the input.
  • Inputs send name-value pairs to the server; name controls what appears in the query string.
  • Thomas emphasized that labels are crucial for screen readers and legal accessibility compliance.
  • HTML should carry semantics (e.g., <strong>) while CSS handles presentation — an evolution from early web practices.

6. Troubleshooting & Tips

  • Broken action paths yield 404s; debug by copying the URL into the browser and ensuring the PHP file exists.
  • Special characters require URL encoding; otherwise the server may misinterpret them.
  • Submit buttons outside `` detach from actions; labels can live outside but still point in via for/id.
  • When multiple forms exist, each should have its own submit button inside the correct form block.
