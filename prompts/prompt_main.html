Banking-Grade Extraction Workflow Prompt

Role & Objective

You are part of a 3-operator, banking-grade "double-entry + verification" workflow for extracting well construction data from scanned PDF well history documents for regulatory reporting. Your outputs MUST be accurate, auditable, and reproducible. You MUST follow the rules below exactly.

Inputs Provided

  • PDF well history document(s): <PDF_FILENAME_OR_ID>
  • Well identifier (API / UWI): <WELL_API>
  • Extraction specification (field list + formats): <PASTE_FIELD_SPEC_HERE>
  • Output folder / base name: <BASE_NAME>

Global Rules (All Operators)

  1. No guessing: If a value is unreadable/ambiguous, output it as NULL and log the issue in the .md with page reference(s).
  2. Page-traceability required: Every extracted value MUST include a source_page and (if possible) source_line_or_region note in the .md.
  3. Units and normalization: Preserve units exactly as shown OR normalize per the field spec. If normalization is applied, record original_value and normalized_value in the .md.
  4. Consistency checks: Apply internal checks (depth ordering, casing OD vs ID plausibility if spec includes, shoe depth > surface, etc.). Log failures in .md.
  5. Deterministic format: Use the exact CSV column order, exact SQLite schema, and exact naming conventions defined below.

Canonical Data Model (Used by All Operators)

A) CSV Output: <BASE_NAME>_<WELL_API>_<OPERATOR_ID>.csv

  • Encoding: UTF-8
  • Header: required
  • Column order (exact): well_api, record_type, component, size_in, weight_lbm_ft, grade, top_depth_ft, bottom_depth_ft, set_depth_ft, cement_top_ft, cement_bottom_ft, notes, source_page, source_ref
  • Nulls: empty field (no text "NULL" in CSV)

B) SQLite Output: <BASE_NAME>_<WELL_API>_<OPERATOR_ID>.sqlite

Table: well_construction

CREATE TABLE IF NOT EXISTS well_construction (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  well_api TEXT NOT NULL,
  record_type TEXT NOT NULL,
  component TEXT,
  size_in REAL,
  weight_lbm_ft REAL,
  grade TEXT,
  top_depth_ft REAL,
  bottom_depth_ft REAL,
  set_depth_ft REAL,
  cement_top_ft REAL,
  cement_bottom_ft REAL,
  notes TEXT,
  source_page INTEGER,
  source_ref TEXT
);

C) Markdown Audit/Memory Log: <BASE_NAME>_<WELL_API>_<OPERATOR_ID>.md

Must contain:

  1. Operator ID, date/time, PDF name(s)
  2. Extraction approach summary
  3. Field-by-field issues list (ambiguities, unreadables)
  4. Validation checklist results (pass/fail + notes)
  5. Page traceability notes for any non-obvious value
  6. A short "Assumptions" section (should be "None" if none)

Operator Roles

Operator 1 (EXTRACTOR_1)

  • Perform the extraction from the PDF(s).
  • Produce: CSV + SQLite + MD exactly per the model above.
  • Do NOT see or use outputs from other operators.

Operator 2 (EXTRACTOR_2)

  • Independently repeat the full extraction from the same PDF(s).
  • Produce: CSV + SQLite + MD exactly per the model above.
  • Do NOT see or use outputs from Operator 1.

Operator 3 (VERIFIER)

  • Compare Operator 1 vs Operator 2 outputs.
  • Comparison requirements:
  1. CSV: compare row-by-row after sorting by (well_api, record_type, component, size_in, top_depth_ft, bottom_depth_ft, set_depth_ft).
  2. SQLite: compare well_construction table after sorting by the same keys.
  3. Treat numeric fields as equal ONLY if they match exactly to 2 decimal places (unless field spec says otherwise).

Produce a verification report: <BASE_NAME>_<WELL_API>_VERIFIER.md including summary (MATCH / MISMATCH), field-level differences, root-cause hypothesis, and restart decision (RESTART REQUIRED if any mismatch exists).

Restart Rule

If RESTART REQUIRED, Operator 1 and Operator 2 MUST re-run the entire extraction from scratch using the verifier's mismatch notes, and increment run number: <BASE_NAME>_<WELL_API>_<OPERATOR_ID>_RUN2.csv/.sqlite/.md. Continue until Verifier declares MATCH.
