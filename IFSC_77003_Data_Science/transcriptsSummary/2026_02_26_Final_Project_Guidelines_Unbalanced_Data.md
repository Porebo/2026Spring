# Lecture Summary: Final Project Guidelines, Unbalanced Data Strategies, and Reporting Expectations
**IFSC 77003 - Data Science & Technologies | February 26, 2026 | Dr. Elizabeth Pierce**

## Session Focus
- Reviewed practical strategies for handling **unbalanced datasets** in rare-event prediction tasks.
- Delivered detailed expectations for the **final data science project** report and 3-minute presentation.
- Clarified rules for **coordinated/team projects**, workload distribution, and documentation of individual contributions.

## 1. Rare Events and Unbalanced Data
- The class connected image analytics topics with broader predictive modeling challenges.
- Dr. Pierce highlighted that high overall accuracy can be misleading when rare classes matter (for example, fraud detection).
- Shared prior resources from Dr. Wu (recording, slides, notebook, dataset) for class-imbalance handling.

### Techniques discussed
- **Oversampling** minority class and **undersampling** majority class.
- Synthetic balancing approaches to improve class representation.
- Loss functions with higher penalties for rare-class errors.
- Combining resampling and weighted-loss strategies.

## 2. Final Project Deliverables and Scale
- Final project is a major grade component designed to demonstrate real data literacy.
- Required deliverables:
  - data science investigation,
  - written report,
  - 3-minute summary video.
- Typical report guideline: **10 to 20 pages** (single-spaced, 12-point, 1-inch margins), with flexibility by project type.

## 3. Individual vs Coordinated Projects
- Students may coordinate around a shared broader problem, but each person is graded individually.
- Each student needs their own:
  - meaningful analysis work,
  - report,
  - presentation,
  - documented hours.
- For teams, expectations scale per person rather than splitting one small effort across multiple names.

## 4. Recommended Report Structure
- Title section (project title, student name, course/term; optional mentor/supervisor context).
- Introduction with motivation, project problem, and objectives.
- Literature review (roughly **10 to 15 sources per person**, higher for teams).
- Methodology/work plan (data sourcing, cleaning, metrics, analysis strategy).
- Analysis and interpretation (explain output clearly, not just screenshots/code dumps).
- Discussion, conclusions, insights, limitations, and future work.
- Appendix with project schedule/hours and references.

## 5. Literature Review and AI-Citation Warning
- Dr. Pierce warned against relying on LLM-generated references without verification.
- Main risk: fabricated or mismatched citations that appear plausible.
- Acceptable source mix includes journal papers, industry reports, blogs, and technical notebooks, as long as sources are valid and properly cited.

## 6. Analysis Expectations
- Students should justify method choices and explain parameter tuning decisions.
- Emphasis on model quality diagnostics: false positives/false negatives, sampling strategy, train/test design, and cross-validation where relevant.
- Strong projects can be exploratory and visualization-heavy if interpretations are rigorous and useful.

## 7. Presentation Guidance (3-Minute Video)
- Goal is concise professional communication, similar to employer-facing project summaries.
- One visual/slide is sufficient; delivery format is flexible.
- Suggested structure: problem, methods, key findings, and contribution.

## 8. Example Projects Discussed
- Housing affordability analysis (exploratory analytics with Tableau dashboards).
- Cybersecurity labor-market supply/demand analysis.
- Discussion of how class projects can scale toward master's or doctoral research.

## 9. Team Project Q&A Clarification
- Teams may split work by region (for example, Northeast/Southwest) or by analysis dimension.
- Shared datasets are acceptable if each student contributes full effort and documents their role.
- Team literature expectations should reflect cumulative effort (for example, 3 students -> around 30 total sources).

## Key Takeaways
1. Rare-event projects require deliberate imbalance handling beyond raw accuracy.
2. Final project quality is judged on both technical work and clarity of interpretation.
3. Coordinated teams must show **individual completeness**, not partial specialization only.
4. Verified references and transparent hours tracking are essential.
5. A concise, well-structured 3-minute presentation is part of professional data communication.
